# Contributing to Output Surface Integrity

Thank you for helping improve Output Surface Integrity (OSI). Keep every contribution bounded, inspectable, and within the repository's current evidence and claim boundaries.

## Welcomed contributions

- documentation corrections;
- lightweight adoption examples;
- schema or template improvements;
- reproducible bug reports;
- evidence or claim-boundary contradictions; and
- bounded integrations that preserve the lightweight path.

Start with an issue before proposing a claim-changing or architecture-changing contribution. Maintainers may reject scope expansion even when the underlying idea is useful.

## Public-data boundary

Do not include secrets, credentials, private logs, personal data, plaintext Gold, plaintext Mapping, Primary Evaluator observations, internal conversation identifiers, or private filesystem paths in issues, pull requests, examples, or attachments. Follow [SECURITY.md](SECURITY.md) for sensitive reports.

Historical evidence must remain intact. Corrections are Forward-only: do not rewrite a past `BLOCK`, adverse result, or unresolved boundary into `PASS`.

## Pull request requirements

Every pull request must state:

- purpose;
- changed paths;
- evidence and validation;
- claim-boundary impact;
- unresolved items;
- rollback or restart point; and
- next safe action and owner.

Use the repository pull request template and keep unrelated changes out of the same contribution.
